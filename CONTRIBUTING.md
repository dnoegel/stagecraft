# Contributing to Stagecraft

Thanks for your interest. The repo is small enough that PRs and issues are both welcome.

## Setup

```bash
git clone <this-repo>
cd stagecraft
npm install
npm run demo          # serves the demo deck at http://localhost:3000
```

## Adding a component

1. Read `AGENT.md` first — it describes the conventions every component must follow (factory pattern, `data-stage-edit`, `data-stage-key`, reveal modes).
2. Read an existing simple component (`src/components/kinetic-text.js` is the canonical pattern).
3. Write your component at `src/components/<kebab-name>.js`.
4. Add CSS to `themes/phosphor/components-<family>.css`. Use CSS variables (`--accent`, `--dim`, `--fg`, etc.) — never hardcode colors.
5. Add an entry in `AGENT.md` with a usage example.
6. Optionally: add a demo slide at `demo/slides/<family>-<N>-<name>.js` and wire it into the manifest.

## Adding a transition

1. Register in `src/transitions.js` via `Stage.registerTransition('name', { enter, exit })`.
2. Add the CSS keyframes in `themes/phosphor/transitions.css`. Other themes inherit via `@import` from phosphor's transitions.
3. Add the picker icon glyph to the `TRANSITION_ICONS` map in `src/edit-mode.js`.
4. Add a brief entry to AGENT.md's transitions table.

## Coding conventions

- **No build step for source.** Authoring is plain ES2020 in `src/`. The `dist/` bundle is generated by `npm run build` and not source-of-truth.
- **One IIFE per file.** Each component / module attaches to `window.Stage` directly. No ES modules at runtime.
- **Theme-aware via CSS variables.** Never hardcode hex colors in component CSS.
- **HTML escape user text.** Every component has a local `escape()` helper for user-provided strings.
- **Cleanup functions returned from init.** Every `init(el)` that starts timers or attaches listeners must return a cleanup function.

## Testing

```bash
npm run verify         # AST roundtrip tests (edit ops)
npm run test:visual    # Playwright visual regression (if configured)
```

## What NOT to add

- Build dependencies beyond what's in `package.json`. Stagecraft's value is no-build-step usage.
- Components that are "almost the same" as existing ones — extend or parametrize instead.
- Heavy graphics libraries (D3, Chart.js, etc.) — Stagecraft is intentionally hand-rolled SVG.

## Filing issues

If you find a bug, a screenshot + the offending slide source code is usually enough. If a transition or component looks visually wrong in a specific theme, name the theme.
